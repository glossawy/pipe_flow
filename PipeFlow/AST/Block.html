Overview

A PipeFlow AST node representing a literal proc (lambda or not). As a destination node, the input is simply passed to the proc it is wrapping and the result is forwarded on.

#reifiable? ⇒ Boolean Also known as: input_needed?

Reifiability determines representability and usability within the context of a pipeline. A parameterized node is reifiable if and only if it, with the metadata available, can reasonably be used as a hole or destination within a pipeline.

Whether or not it is possible to use the proc in a pipeline. There must be exactly one required parameter OR zero required parameters and at least one optional parameter.

Examples:

Reifiable procs

->(x) { ... }                  # One required parameter
->(x, y = 2, z: 3) { ... }     # Many parameters, only one is required
proc { |x, y, z, *args| ... }  # all parameters are optional in a proc

Non-reifiable proc

# two required parameters
->(x, y) { ... }

def reifiable?
  arity.min == 1 ||
    (arity.min.zero? && arity.max > 0)
end

#to_definition ⇒ String

Attempts to derive a sane string representing a likely definition for the parameterized object.

Examples:

Proc

block = AST::Block.new(proc { |x, y = 2| x + y })
block.definition  # => "proc { |x, y = <value>| ... }"

Lambda

block = AST::Block.new(->(x, y = 2, *args, z:) { x + y + args.sum - z })
block.definition  # => "->(x, y = <value>, *args, z:) { ... }"

    derived_arity: arity,
